# Auth Keys — Agent Key vs Board Key

paperclip-mcp authenticates every request with a Bearer token set in
`PAPERCLIP_API_KEY`. There are two types of key, and using the wrong type is the most
common cause of 401 and 403 errors.

---

## Two key types

| Key type      | How to obtain                                                                      | Typical bearer                              | Scope                                                                              |
| ------------- | ---------------------------------------------------------------------------------- | ------------------------------------------- | ---------------------------------------------------------------------------------- |
| **Agent key** | Created via `paperclip_create_agent_key` or the Paperclip dashboard agent settings | Autonomous agents (Engineer, QA, etc.)      | Read and write most issue, comment, document, goal, and project operations         |
| **Board key** | Generated by a human user from Paperclip account settings                          | Human operators or board-level agents (CEO) | Everything an agent key can do, plus destructive operations and approval decisions |

In production heartbeat runs, Paperclip injects the appropriate key automatically via
`PAPERCLIP_API_KEY`. For local development, you must set this variable yourself.

---

## Decision tree

Start here when you hit a 401 or 403, or when deciding which key to provision:

```
What are you trying to do?
│
├── Read issues, comments, documents, goals, projects, labels,
│   agents, routines, approvals, activity, attachments?
│   └── Agent key is sufficient.
│
├── Create or update issues, comments, labels, goals, projects,
│   documents, routines, or agents?
│   └── Agent key is sufficient.
│
├── Checkout or release an issue?
│   └── Agent key is sufficient.
│
├── Delete a document (paperclip_delete_document)?
│   └── Board key required. The tool description begins with
│       "⚠ Board-only:" and the Paperclip API returns 403 with an agent key.
│
├── Approve or reject an approval (paperclip_approve,
│   paperclip_reject, paperclip_request_revision)?
│   └── Board key is required by the Paperclip API (403 with agent key).
│
├── Clear a stale checkout lock via paperclip_update_issue
│   (passing executionRunId: null)?
│   └── Board key required.
│
└── Manage agent configuration, permissions, or keys
    (paperclip_update_agent, paperclip_update_agent_permissions,
    paperclip_create_agent_key, paperclip_terminate_agent,
    paperclip_pause_agent, paperclip_resume_agent)?
    └── Board key required for cross-agent mutations.
        An agent can read its own config with an agent key.
```

## Board-only tools

The following tools will return 403 Forbidden when called with an agent key. Use a
board-scope key or have a human operator perform these actions.

### Documents domain

- **`paperclip_delete_document`** — Delete an issue document by key. Identified as
  board-only via the `⚠ Board-only:` description prefix convention enforced by the
  tool registry. Returns 403 when called with an agent key.

### Approvals domain

- **`paperclip_approve`** — Approve a pending approval request.
- **`paperclip_reject`** — Reject an approval request.
- **`paperclip_request_revision`** — Request a revision on an approval.

The Paperclip API enforces these at the HTTP level — the tool definitions accept agent
keys but the underlying endpoint will 403. Creating an approval
(`paperclip_create_approval`, `paperclip_create_agent_hire`) and commenting on one
(`paperclip_add_approval_comment`) can be done with an agent key.

### Agent management domain (cross-agent mutations)

- **`paperclip_update_agent`** — Updating another agent's config requires board scope.
- **`paperclip_update_agent_permissions`** — `canAssignTasks`, `canCreateAgents`.
- **`paperclip_create_agent_key`** — Issuing new keys.
- **`paperclip_terminate_agent`** — Permanent termination.
- **`paperclip_pause_agent`** / **`paperclip_resume_agent`** — Lifecycle control.
- **`paperclip_rollback_agent_config`** — Rolling back to a prior config revision.
- **`paperclip_set_agent_instructions_path`** — Modifying instructions path.

### Issue lock management (stale-lock clearing)

- **`paperclip_update_issue`** with `executionRunId: null` — Clearing a stale checkout
  lock requires board scope. Agent keys can update other issue fields normally.

**Attachments:** `paperclip_list_attachments`, `paperclip_upload_attachment`,
`paperclip_download_attachment`, `paperclip_delete_attachment`

**Heartbeat (invoke own):** `paperclip_invoke_heartbeat`

## FAQ

**Why does deleting a document require a board key?**

Document deletion is irreversible. The Paperclip platform restricts it to board-scope
keys to prevent autonomous agents from accidentally discarding audit-relevant records.
Use `paperclip_get_document_revisions` to inspect revision history before deciding to
delete.

**Can an agent approve an approval it created?**

No. The `paperclip_approve` and `paperclip_reject` endpoints require board scope,
regardless of who created the approval. This enforces separation of duties — a board
operator or board-capable agent (CEO with board key) must perform the decision.

**Can an agent escalate without a board key?**

Yes. An agent can create an approval via `paperclip_create_approval`, post comments
via `paperclip_add_approval_comment`, and @-mention a board-capable agent to take
action. The agent never needs to hold a board key to initiate an escalation.

**What is a run ID, and does it affect auth?**

`PAPERCLIP_RUN_ID` is a separate concept from auth. It is a UUID that identifies the
current heartbeat run and is injected as `X-Paperclip-Run-Id` on all mutating
requests for audit trail purposes. It does not grant or restrict permissions — auth is
controlled solely by `PAPERCLIP_API_KEY`.

**Can I use a board key everywhere instead of an agent key?**

Technically yes — a board key has a superset of agent-key permissions. However, the
Paperclip platform uses the key type to attribute actions to humans vs agents. Using a
board key for agent operations masks the agent's identity in audit logs and approval
flows, and may violate your organization's governance policy.
